Architecture Report: experiments

Date: 2026-05-12

Do not implement yet. This Phase 1 report identifies deepening opportunities only. No production code was changed.

Review Scope

Read repo-local instructions, README, compact tree output, and sampled the reusable tensor_logic package, proof/search Modules, experiment scripts, and tests. No ADR conflicts were found.

The review uses the architecture vocabulary: Module, Interface, Implementation, Depth, Shallow, Seam, Adapter, Leverage, and Locality.

1. Make Rule Parsing And Rule Execution One Deep Module

Files: tensor_logic/program.py, tensor_logic/rules.py, tensor_logic/proofs.py, tensor_logic/provenance.py, tests/test_tensor_logic_core.py, tests/test_reason.py

Problem: The rule Interface is split across multiple grammars and execution paths. The caller must understand parser shape, arity limits, negation support, and proof compatibility.

Solution: Deepen a rule-front-end Module with one Rule representation and adapters for tensor evaluation, proof search, and provenance.

Benefits: Better Locality for grammar changes and higher Leverage for callers. The deletion test suggests the current split is shallow because deleting one parser would move complexity into another caller.

Test Impact: Parser, AST, proof, and provenance behavior become testable through one Interface.

Risk: Moderate; experiments may rely on quirks in both current grammars.

Suggested Validation: python -m pytest tests/test_tensor_logic_core.py tests/test_reason.py tests/test_proof_recursion.py -v

2. Deepen Proof Search Around A Proof Engine Seam

Files: tensor_logic/proofs.py, tensor_logic/proof_result.py, tensor_logic/proof_tree_viewer.py, tensor_logic/execution.py, tensor_logic/http_api.py

Problem: Positive proof, negative proof, recursion, interventions, and formatting leak across callers. The Interface exposes much of the Implementation.

Solution: Add a proof engine Module whose Interface is goal plus options to typed proof result. CLI, HTTP, and web become adapters.

Benefits: Better Locality for proof bugs and more Leverage for external callers.

Test Impact: Tests assert proof-result semantics through the proof engine seam.

Risk: Moderate; binary-relation assumptions are widespread.

Suggested Validation: python -m pytest tests/test_proof_recursion.py tests/test_reason.py tests/test_web_workbench.py -v

3. Extract A Reusable Experiment Harness

Files: experiments/exp79_self_play_loop.py, experiments/exp81_sweep.py, experiments/exp81_optimize_rule_induction.py, experiments/exp83_slot_attention.py, tensor_logic/research/utils.py

Problem: Experiment scripts repeat lifecycle concerns, so each script's Interface is shallow and reproduction knowledge is scattered.

Solution: Put config, deterministic seeding, result paths, quick/full modes, and metric summaries behind a research harness Module.

Benefits: Higher Locality for run behavior and higher Leverage for future experiments.

Test Impact: Harness tests can prove deterministic metadata; experiment tests stay focused.

Risk: Low to moderate; keep the seam light because this is research code.

Suggested Validation: python -m pytest tests/test_exp79.py tests/test_exp81.py tests/test_exp83_slot_attention.py -v
